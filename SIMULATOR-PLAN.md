who ride alongside the user on ImagineFun rides, wearing real captured cosmetics.

## Step 5: Ride-vehicle catalog + seat learning

Goal: replace the heuristic `findEmptySeats` with **explicit seat profiles per
ride-vehicle**, learned from real mount sessions and labeled by hand.

### Background: the relevant geometry

ImagineFun rides only spawn an actual mountable armor stand for **occupied** seats.
Empty seats expose only persistent placeholder stands at +0.32y above the seat
platform. A seat profile lets the simulator render ghosts at the correct position
without needing a server-spawned mount stand for that seat.

### Three-stage data flow

```
              user curates                                user labels in
              by hand                                     Claude Code
              │                                           │
              ▼                                           ▼
       ┌──────────────┐         ┌─────────────────┐   ┌────────────┐
       │ vehicles.    │         │ seat-samples.   │   │ seats.json │
       │   json       │ ──────▶ │   json          │ ─▶│ (curated)  │
       └──────────────┘         └─────────────────┘   └────────────┘
       (recognition criteria)   (per-mount samples,    (clustered &
                                 written by mod)       labeled)

       Mod reads       Mod writes        Mod reads (consume)
```

### File: `vehicles.json` (you curate, mod reads)

Path: `<configDir>/imaginemorefun/simulator/vehicles.json`

Keyed by `RideName.getShortName()` (e.g. `mainst`, `dlrr`, `storybook`).

```json
{
  "rides": {
    "mainst": {
      "vehicles": [
        {"id": "main_street_omnibus",            "match": {"itemId": "minecraft:diamond_pickaxe", "damage": 442}},
        {"id": "main_street_horseless_carriage", "match": {"itemId": "minecraft:golden_shovel",   "damage": 1554}}
      ]
    },
    "dlrr": {
      "vehicles": [
        {"id": "dlrr_locomotive",     "match": {"itemId": "minecraft:diamond_pickaxe", "damage": 100}},
        {"id": "dlrr_passenger_car",  "match": {"itemId": "minecraft:netherite_axe",   "damage": 12}}
      ]
    }
  }
}
```

The runtime matches a ride's candidate vehicles against nearby decoration
armor stands by their head-slot item id + damage value.

### File: `seat-samples.json` (mod writes, you read)

Path: `<configDir>/imaginemorefun/simulator/seat-samples.json`

```json
{
  "samples": [
    {
      "ride": "mainst",
      "vehicleId": "main_street_omnibus",
      "anchorPos": [10.62, 64.88, 190.65],
      "anchorYaw": -248.4,
      "userSeatLocalOffset": [-1.34, -0.12, 0.32],
      "yawDelta": 0.0,
      "ticks": 142,
      "ts": 1777456789000
    }
  ]
}
```

One entry per mount-session — the median local offset across all ticks of that sit.

### File: `seat-clusters.json` (mod writes, you label in Claude Code)

Generated by `/imf-sim cluster`. You paste it into Claude Code; we fill in `label`
fields together; I write the labeled version to `seats.json`.

```json
{
  "main_street_omnibus": {
    "ride": "mainst",
    "clusters": [
      {"centroid": [-1.34, -0.12, 0.32], "yawDelta": 0.0, "sampleCount": 3, "label": null},
      {"centroid": [+0.66, -0.12, 0.32], "yawDelta": 0.0, "sampleCount": 2, "label": null}
    ]
  }
}
```

### File: `seats.json` (curated, mod reads at runtime)

Path: `<configDir>/imaginemorefun/simulator/seats.json`

```json
{
  "main_street_omnibus": {
    "seats": [
      {"label": "lower_deck_left",  "localOffset": [-1.34, -0.12, 0.32], "yawDelta": 0.0},
      {"label": "lower_deck_right", "localOffset": [+0.66, -0.12, 0.32], "yawDelta": 0.0}
    ]
  }
}
```

## Implementation phases

| Phase | What | Cost | Status |
|---|---|---|---|
| **A** | `vehicles.json` schema + loader (`VehicleCatalog`) | small | in progress |
| **B** | `/imf-sim vehicle-debug` + `/imf-sim list-vehicles` | trivial | in progress |
| C | `SeatLearner` tick loop (anchor detection, per-tick local-offset sample, per-mount commit) | medium | next |
| D | `/imf-sim learn on/off` + persisted `seat-samples.json` | small | next |
| E | `/imf-sim cluster` + `seat-clusters.json` writer | medium | later |
| F | Read `seats.json`, replace `findEmptySeats` for known vehicles | small | later |

Each phase is shippable on its own. A+B is the smallest useful slice — you can
identify vehicles before the learning loop is wired up.

## Workflow once A+B ship

1. Get on a vehicle (e.g. Main Street omnibus).
2. Run `/imf-sim vehicle-debug`.
   - If we have a vehicle registered for this ride that matches → it'll print the
     matched id + anchor stand.
   - Otherwise it lists every nearby armor stand's head-slot item so you can
     pick which item id + damage represents the cart.
3. You hand me `(rideShortName, vehicleId, itemId, damage)`. I append to `vehicles.json`.
4. Restart Minecraft (or `/imf-sim reload`, future).
5. `/imf-sim vehicle-debug` should now show the matched id.

Repeat per ride+vehicle combo. The catalog grows incrementally.

## Known caveats

- **Per-segment respawn**: on some rides the visible decoration stands respawn
  mid-ride alongside mount stand changes. The vehicle anchor entity id will
  change. Match-by-criteria (not by id) handles this transparently.
- **Multiple vehicles in a ride**: dlrr has many cars. Recognition by item id
  is per-vehicle-type, not per-instance — both passenger cars match the same
  vehicle id, and the seat profile applies identically. Good enough for now.
- **`damage` is the resource-pack model selector** on ImagineFun. Different
  vehicles distinguished by different damage values on the same base item are
  fine.
